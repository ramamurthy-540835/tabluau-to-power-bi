Executive SummaryThis document specifies the design, architecture, and build plan for a Tableau to Power BI Migration Accelerator. The accelerator's mission is to compress migration timelines from 9–18 months (manual) to 3–6 months by automating the deterministic 60–70 percent of work and intelligently assisting the judgment-heavy 30–40 percent.The accelerator follows a hybrid architecture. A deterministic Python core handles parsing, lineage, schema mapping, and code generation. An LLM-assisted layer (Claude or equivalent) handles semantic translation tasks where pattern matching is insufficient — primarily Tableau LOD expressions to DAX, calculated field translation, and dashboard layout reasoning. This separation is deliberate: deterministic code for things that must be exact, AI for things that require judgment.1.1 ScopeIn scope: Tableau Desktop (.twb, .twbx), Tableau Server / Cloud workbooks, published data sources (.tds, .tdsx), Tableau Prep flows (.tfl, .tflx), and the full translation pipeline to Power BI Desktop (.pbix), Power BI Project (.pbip), TMDL semantic models, and Power Query M scripts.Out of scope (Phase 1): Tableau Pulse insights, Einstein Discovery models, Tableau Bridge configurations, custom JavaScript API embeddings, and viz extensions written in custom code.1.2 Success MetricsInventory accuracy: 100% of artifacts catalogued with complexity scoring.Automated translation coverage: ≥70% of calculated fields, ≥85% of visuals, ≥90% of data source connections.Effort reduction: ≥60% reduction in person-hours versus baseline manual migration.Output quality: ≥95% of generated PBIX files open without errors; ≥80% require no rework on first review.2. Problem Statement and First Principles2.1 Why Migrations FailTableau-to-Power BI migrations fail for one structural reason: the two platforms have fundamentally different mental models. Tableau is viz-first (build worksheets, then assemble dashboards). Power BI is model-first (build a semantic model, then visuals consume it). A naive 1:1 translation produces functional but architecturally incorrect Power BI reports — flat tables instead of star schemas, calculated columns instead of measures, no separation of fact and dimension. These reports work but do not scale and do not perform.2.2 First-Principles DecompositionThe migration problem decomposes into seven independent layers, each with its own translation logic:Connectivity layer — data source connections and credentials.Data preparation layer — Tableau Prep flows to Power Query M.Semantic layer — physical model, relationships, hierarchies.Calculation layer — calculated fields, LODs, table calcs to DAX.Visualization layer — worksheets to visuals.Interaction layer — filters, parameters, actions to slicers, drillthrough, bookmarks.Distribution layer — Sites/Projects to Workspaces/Apps, security, schedules.The accelerator processes these layers in dependency order. Each layer has a typed input contract and a typed output contract. This means any layer can be replaced or upgraded independently — including swapping the LLM provider.2.3 The 70/30 RuleEmpirical observation across migrations: roughly 70% of translation work is deterministic pattern matching (connection strings, field mappings, basic aggregations, standard chart types). Roughly 30% requires semantic understanding — LOD expressions, complex calculated fields, blended data sources, dashboard intent. The accelerator automates the 70% completely and assists the 30% with LLM-driven suggestions plus human review queues.3. High-Level Architecture3.1 Architectural PrinciplesPipeline, not monolith. Each module is a pure function with typed input and output. Modules are composable and independently testable.Intermediate Representation (IR). All translation passes through a canonical IR. Tableau parsers emit IR; Power BI generators consume IR. This decouples source and target — a future Looker or Qlik parser would plug into the same IR.Deterministic core, AI edges. AI is used only where deterministic rules cannot reach acceptable quality. Every AI call is logged, versioned, and replayable.Idempotent and resumable. Re-running the accelerator on the same input produces the same output. Long-running jobs checkpoint after each module.Human-in-the-loop by design. The accelerator always produces a Review Queue alongside the output, flagging items where confidence is below threshold.3.2 Component DiagramThe system has six top-level Detailed Module Specification4.1 Module: IngestionPurposeAccept Tableau artifacts in any supported format and normalize them to raw XML/JSON for parsing.Inputs.twb (XML), .twbx (ZIP containing .twb plus extracts), .tds, .tdsx, .tfl, .tflx, .hyper extracts, Tableau Server REST API endpoints with auth token.OutputsNormalized RawArtifact object: { artifact_id, type, raw_xml, embedded_extracts[], metadata }.Key behaviorsUnzip .twbx and .tdsx using Python zipfile; preserve embedded extracts in temp storage.For Server/Cloud sources, use Tableau REST API to download workbooks, data sources, and metadata. Respect rate limits (default 200 req/min).Compute SHA-256 of source file for deduplication and idempotency tracking.Detect file format version (Tableau 9.x through 2024.x) and route to appropriate parser variant.Edge casesPassword-protected .twbx — fail with clear error, do not attempt decryption.Workbooks with embedded credentials — strip credentials in IR, log warning, never persist secrets.Hyper extracts >10GB — stream-process; do not load into memory.4.2 Module: ParserPurposeConvert raw Tableau XML into the typed Intermediate Representation.Sub-parsersSub-parserTableau XML elementOutput IR typeDataSourceParser<datasource>IRDataSource (connections, columns, joins, calculations)WorksheetParser<worksheet>IRWorksheet (rows, cols, marks, encodings, filters)DashboardParser<dashboard>IRDashboard (zones, layout, actions, sizing)StoryParser<story>IRStory (storypoints, captions)ParameterParser<datasource><column param>IRParameter (type, allowable values, current)FormulaParser<calculation formula>IRFormulaAST (recursive descent over Tableau formula grammar)ActionParser<action>IRAction (filter, highlight, URL, set actions)StyleParser<style>, <format>IRStyle (fonts, colors, number formats)Critical: the FormulaParserThis is the single highest-value parser. Tableau's formula language has approximately 280 functions across logical, string, date, aggregate, table calc, and LOD categories. The parser must produce a typed AST, not just tokenize. Reasons:LOD expressions ({ FIXED [Region] : SUM([Sales]) }) require semantic understanding of the dimensionality scope — not lexical translation.Table calcs (WINDOW_SUM, RUNNING_SUM, INDEX) compute over partition/addressing — must be reified as DAX window functions or rewritten as measures.Type inference is needed because Tableau infers types implicitly; DAX requires explicit handling for INT vs DECIMAL vs CURRENCY.Recommended grammar approachUse Lark or PyParsing to define an EBNF grammar for Tableau formulas. Avoid regex — formulas nest arbitrarily.AST node types: Literal, ColumnRef, ParameterRef, FunctionCall, LODExpression, IfExpr, CaseExpr, AggregateCall, TableCalcCall, BinaryOp, UnaryOp.Maintain a function dictionary with arity, return type, and a 'translation strategy' field (direct, rewrite, llm-assisted).4.3 Module: Intermediate RepresentationThe IR is the contract between everything upstream and everything downstream. Get this right and the rest follows.Top-level structureIRMigrationUnit├── metadata: { source_version, parsed_at, source_hash, ... }├── data_sources: List[IRDataSource]│   ├── connections: List[IRConnection]│   ├── tables: List[IRTable]│   ├── joins: List[IRJoin]│   ├── columns: List[IRColumn]   # physical + calculated│   ├── relationships: List[IRRelationship]   # inferred│   └── filters: List[IRFilter]   # data source filters├── parameters: List[IRParameter]├── worksheets: List[IRWorksheet]│   ├── used_data_source: ref│   ├── shelf_assignments: { rows, cols, color, size, label, detail, tooltip }│   ├── mark_type: enum│   ├── filters: List[IRFilter]│   ├── sort: IRSort│   └── format: IRStyle├── dashboards: List[IRDashboard]│   ├── size: { width, height, mode }│   ├── zones: List[IRZone]   # tree, not flat — zones nest│   ├── actions: List[IRAction]│   └── parameter_controls: List[ref]└── analysis: IRAnalysisResults  # populated by Analyzer moduleDesign rulesEvery IR node has a stable id (UUID) so cross-references survive serialization.IR is fully serializable to JSON. This enables disk caching, replay, and debugging.IR uses Pydantic models with strict type validation. Bad parser output fails at construction, not deep in the pipeline.Backward compatibility: IR schema is versioned. Future schema changes go through migrations, never silent breakage.4.4 Module: AnalyzerPurposeCompute lineage, complexity scores, dead-code detection, and migration risk metrics on the IR. This module produces the inputs that drive the rationalization phase of the migration.Sub-analysesAnalysisWhat it computesField lineageFor every calculated field, the dependency graph back to physical columns. Used to detect orphaned calcs and to order translation.Worksheet usageWhether a worksheet is referenced by any dashboard or story. Unused worksheets are candidates for omission.Calculation complexity scoreComposite score 0–100 based on: AST depth, LOD presence, table calc presence, function category, parameter dependency. Drives routing between deterministic and LLM translators.Dashboard complexity scoreComposite based on: zone count, action count, cross-data-source actions, custom layouts (tiled vs floating mix).Data source overlapHash-based detection of data sources that share underlying tables. Used to consolidate into a single Power BI semantic model.Anti-pattern detectionFlags: blended data sources (must convert to model relationships), filter-as-parameter hacks, calculated fields that should be measures, hard-coded date filters, etc.Migration risk score0–100 score per artifact, combining the above. Drives prioritization in the migration roadmap.4.5 Module: Deterministic TranslatorsThese translators handle the 70% that is rule-based. Each is a pure function from IR-fragment to Power BI artifact-fragment.4.5.1 Connection TranslatorMaps Tableau connection types to Power Query connectors. Maintain an explicit connection_map.yaml keyed on Tableau connection class (sqlserver, oracle, snowflake, redshift, postgres, mysql, googlesheets, excel-direct, etc.).Generates Power Query M source steps, including server, database, schema, and authentication mode placeholders.Handles Tableau's 'connection' vs 'named-connection' nesting. Outputs one M Source per logical connection.4.5.2 Schema TranslatorTranslates physical tables and joins from Tableau data model into Power BI relationships.Detects star-schema candidates: any table referenced by multiple worksheets through GROUP BY-style aggregation is a fact; tables joined to it on a single key are dimensions.Generates relationship cardinality and cross-filter direction. Default: many-to-one, single direction. Flag bidirectional only when explicit blending requires it.Emits TMDL relationship declarations with proper isActive flags for ambiguous paths.4.5.3 Visual TranslatorMaps Tableau mark types to Power BI visual types via a mark_visual_map.yaml (bar→clusteredBarChart, line→lineChart, scatter→scatterChart, map→azureMap, gantt→customGantt, etc.).Maps shelf encodings: Rows/Columns→Axis, Color→Legend, Size→Size, Label→Data labels, Detail→implicit groupings.Generates visual JSON in PBIP format with correct binding to fields.For visuals with no direct equivalent (treemap-with-bullet-chart, custom Tableau extensions), routes to Deneb Vega-Lite generator or flags for manual.4.5.4 Filter and Parameter TranslatorTableau quick filters → Power BI slicers with same dimension.Tableau parameters → Power BI what-if parameters (numeric) or single-select tables (string) — choice driven by parameter usage in calcs.Tableau context filters → no direct equivalent; emit DAX measure variants using CALCULATE with explicit filter context.Tableau sets — fixed sets translate to slicer-bound measures; computed sets translate to DAX-defined groups.4.5.5 Dashboard Layout TranslatorTableau dashboards use absolute pixel coordinates. Power BI uses 1280x720 default canvas with grid snapping.Translation algorithm: scale all coordinates by min(targetW/sourceW, targetH/sourceH); preserve relative positions; snap to nearest 10px grid.Floating zones map to overlapping visuals (not recommended); tiled zones map to grid-positioned visuals (preferred).Dashboard actions: filter actions→cross-filter or drillthrough; URL actions→action buttons with web URL; highlight actions→cross-highlight (default behavior, no extra config).4.6 Module: LLM-Assisted TranslatorsThis is where AI earns its keep. These translators handle inputs where deterministic rules produce >5% error rate.4.6.1 Calculated Field to DAX TranslatorRouting logicEach calculated field is first attempted through the deterministic AST-based translator. The deterministic translator handles ~60% of cases (simple aggregations, basic IF/CASE, string functions, date functions). Failures and complex cases route to LLM.Triggers for LLM routingAST contains LOD expression with non-trivial dimensionality.AST contains nested table calculations.AST depth > 6.AST references >3 calculated fields (composition complexity).Deterministic translator returned with confidence < 0.85.Prompt designThe prompt to Claude includes:System prompt: role definition (DAX expert with deep Tableau knowledge), output format contract (JSON with fields: dax_expression, measure_or_column, confidence, rationale, caveats).Few-shot examples: ~12 curated translation pairs covering each LOD type, table calc, and common pattern.Context: the IR fragment for this calc, plus the IR fragments for any calcs it depends on, plus the relevant table schema.Instruction: explicit guidance on measure-vs-calculated-column choice (default to measure unless row-level aggregation is required).Validation loopGenerated DAX is syntactically validated using Tabular Editor 2 CLI.On syntax failure, retry with the error message appended to the prompt (max 2 retries).On persistent failure, route to Review Queue with original Tableau formula, attempted DAX, and error.Successful translations are also semantically validated when possible: execute on a sample dataset, compare aggregate against Tableau's value.4.6.2 Dashboard Intent TranslatorFor dashboards flagged as 'high complexity' by the analyzer, an LLM pass infers the intent (executive summary vs operational vs exploratory) and proposes a Power BI page structure that better fits Power BI conventions.This is suggestive, not authoritative — output goes into the Review Queue with both the literal translation and the redesign proposal.4.6.3 Naming and Documentation GeneratorTableau workbooks often have terse or cryptic field names ('Calc1', 'Sales 2'). LLM generates suggested business-friendly names and descriptions.Outputs go into the TMDL model as descriptions and into the Review Queue for human approval before commit.4.6.4 Cost and Latency ControlsPrompt caching enabled for system prompt and few-shot examples (Anthropic prompt caching).Batch API used for non-interactive bulk runs — 50% cost reduction.Token budget per workbook tracked and reported. Hard cap configurable (default $5/workbook).Aggressive caching: hash of (formula AST + context schema) → translation result. Re-runs hit cache.4.7 Module: Power BI Generators4.7.1 TMDL Semantic Model GeneratorEmits the model.tmdl, tables/*.tmdl, relationships.tmdl, and cultures.tmdl files in PBIP format.TMDL is text-based, line-oriented, and Git-friendly. Generation is straightforward template emission with proper indentation handling.Handles measures, calculated columns, calculation groups, perspectives, hierarchies, KPIs, and translations.4.7.2 Power Query M GeneratorEmits one .m file per query. Uses canonical step naming (Source, ChangedType, FilteredRows, AddedColumn, etc.).Translation of Tableau Prep flows: each Prep node becomes one or more M steps. Clean nodes become Table.TransformColumns; Pivot nodes become Table.Pivot/Unpivot; Aggregate nodes become Table.Group.Privacy levels and query folding annotations are added based on connection type.4.7.3 Report (Visual JSON) GeneratorEmits report.json in PBIP format. Each page is a section with visualContainers.Each visual is a JSON object with config (visualization type, query, projections), filters, format, and position.Theme JSON generated from Tableau's color palette and font settings, mapped to Power BI's theme schema.4.7.4 PackagingPBIP folder structure assembled with the generated TMDL, M, and JSON files.Optionally compiled to .pbix using pbi-tools for backwards-compatible delivery.Output is validated by opening with Power BI Desktop in headless mode (where infrastructure permits).5. Key Algorithms5.1 LOD Expression TranslationLOD expressions are the hardest single problem in this migration. The translation rules are:Tableau LODDAX equivalent patternNotes{ FIXED [Dim] : AGG([Measure]) }CALCULATE(AGG([Measure]), ALLEXCEPT(Table, Table[Dim]))FIXED ignores all visual context except the named dimensions.{ INCLUDE [Dim] : AGG([Measure]) }CALCULATE(AGG([Measure]), VALUES(Table[Dim]))INCLUDE adds dimensions to the existing context.{ EXCLUDE [Dim] : AGG([Measure]) }CALCULATE(AGG([Measure]), ALL(Table[Dim]))EXCLUDE removes the named dimension from context.{ FIXED : AGG([Measure]) }CALCULATE(AGG([Measure]), ALL(Table))Empty FIXED scope = grand total.Nested LOD: { FIXED [A] : { FIXED [B] : ... } }Multi-step CALCULATE with named variablesGenerate using VAR statements for readability.5.2 Table Calculation TranslationTableau table calcs operate on the visual's row order and partitioning. DAX achieves equivalent results via window functions or measure-based logic over visual queries.Tableau functionDAX strategyCaveatRUNNING_SUM(SUM([X]))CALCULATE(SUM([X]), FILTER(ALLSELECTED(DateTbl), DateTbl[Date] <= MAX(DateTbl[Date])))Requires explicit ordering dimension.WINDOW_AVG(SUM([X]), -3, 0)AVERAGEX over date range with offsetDefine window via DATESBETWEEN or relative date filter.INDEX()RANKX(ALLSELECTED(...))Visual-context dependent; safer to expose as measure.FIRST() / LAST()EARLIEST / LATEST in iterationOften unnecessary in DAX — review before translation.LOOKUP([X], -1)OFFSET(-1, ORDERBY(...)) [Power BI 2024+]OFFSET is a recent DAX function; check tenant version compatibility.5.3 Star Schema Inference AlgorithmTableau workbooks often combine flat tables and joins without an explicit dimensional model. The accelerator infers the star schema:Build a graph where nodes are tables and edges are joins.For each table, compute fan-in (how many other tables join TO it) and fan-out (how many it joins to).Tables with high fan-in and used as group-by sources in worksheets are facts.Tables with low fan-in and high fan-out (typically lookup-shaped) are dimensions.Bridge tables (high fan-in and high fan-out) are flagged for manual review — they often indicate many-to-many relationships requiring careful handling.Date columns across multiple tables are consolidated into a single shared Date dimension.The output is a proposed star schema, presented in the migration report for human confirmation before TMDL generation.5.4 Dashboard Layout ConversionTableau dashboards may be fixed size, automatic, or range. Power BI pages are fixed size (default 1280×720) with optional dynamic sizing. Algorithm:Read Tableau dashboard dimensions; if 'automatic', use the stored last-rendered dimensions.Compute scale factor: min(1280/sourceW, 720/sourceH).For each zone, scale x, y, w, h by scale factor.Snap to 10px grid to avoid sub-pixel positioning.Detect overlapping zones (Tableau allows overlap; Power BI handles it but it's discouraged) and emit a warning in the Review Queue.Filter zones, parameter controls, and legends are extracted and re-positioned as Power BI slicers in a left-side panel by convention.6. Non-Functional Requirements8.1 PerformanceSingle workbook (typical, 5–15 worksheets, 30–80 calculated fields): end-to-end < 90 seconds.Inventory of 1,000 workbooks: < 30 minutes.Bulk translation of 100 workbooks: < 4 hours with LLM batching.LLM call concurrency: configurable, default 8 parallel calls per worker. Workers scale horizontally.8.2 ReliabilityEach module is idempotent. Re-running with the same input and config produces the same output.Checkpointing after each module. A crash mid-pipeline resumes from last checkpoint, never from scratch.All LLM calls are logged with full request and response. The dataset of calls is replayable.Output validation gate: a generated PBIP that fails to compile blocks the run from being marked complete.8.3 SecurityCredentials are never persisted in IR or output. Connection strings are templated; secrets are externalized.LLM calls use Anthropic Workspaces with PII redaction enabled. Sensitive column values (sample data) are never sent.All artifacts encrypted at rest. Customer data is segregated by tenant.RBAC at the API layer when productized. Audit log of every translation decision retained for 12 months.8.4 ObservabilityStructured logging (JSON) at every module boundary.Metrics: parse success rate, translation confidence distribution, LLM token usage, time per module, review queue depth.Per-run migration report with executive summary, module-by-module stats, and review queue export.8.5 ExtensibilityNew source platform (Looker, Qlik, Cognos): add a new parser that emits IR. Everything downstream works unchanged.New target platform (Tableau backwards, Looker Studio): add a new generator that consumes IR.New LLM provider: implemented via the LLMProvider interface; swap by config change.Custom function translations: extend function_dictionary.yaml — no code change required.9. Build Approach: Python vs LLM-DrivenYou asked whether to build this with Python or with an AI like Claude. The honest answer: both, in well-defined roles. A pure Python build is brittle on the calculation translation layer. A pure LLM build is non-deterministic, expensive, and unverifiable on the parsing and generation layers. The right architecture splits the work.9.1 What Python Does Well HereTaskWhy Python winsXML parsing of .twb/.twbxDeterministic, well-defined schema, lxml is mature and fast.AST construction of formulasGrammar is finite. LLM cannot reliably do graph algorithms at scale.Validation and packagingCalls to external tools (Tabular Editor, pbi-tools) and file I/O.9.2 What an LLM Does Well HereTaskWhy an LLM winsLOD/table calc to DAX translationRequires semantic understanding of intent. LLM proposes; human confirms.Dashboard intent inferenceInferring whether a dashboard is exec-summary vs operational vs exploratory.Documentation generationWriting measure descriptions, table descriptions, model documentation.9.3 Recommended Build SequenceBuild the deterministic core first. Testing Strategy10.1 Unit TestingEvery parser sub-module has golden-file tests: input XML fragment → expected IR JSON. ~500 fixtures across function categories.Every deterministic translator has input/output fixtures.Pydantic schema validation runs in CI on every fixture.10.2 Integration TestingEnd-to-end test corpus: 50 representative workbooks spanning easy, medium, hard complexity.For each, expected outputs are manually validated once and become the golden baseline.Regression detection on every PR: any change in output for the corpus must be explicitly approved.10.3 LLM Output EvaluationLLM-as-judge: Claude Opus evaluates each translated DAX measure against the original Tableau formula on a 5-point semantic equivalence scale. Used as a leading indicator.Numerical equivalence test: where a sample dataset is available, execute both the Tableau formula (via Tableau Hyper API) and the generated DAX (via a test Power BI model) and compare results.Human review sample: 10% of all LLM-translated outputs are routed to a BI engineer for blind review. Tracked over time as the ground-truth quality metric.10.4 Acceptance TestingFor each customer engagement, define a 'gold workbook' set — typically 5–10 workbooks the customer considers representative. Migration is accepted only when these pass review.UAT script: report opens cleanly in Power BI Desktop; refresh succeeds; visual outputs match Tableau within tolerance; performance acceptable under expected data volumes.11.
